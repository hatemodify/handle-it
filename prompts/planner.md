## PRD 작성 형식

### 1. 개요
- **한 줄 설명**: 엘리베이터 피치 (15단어 이내)
- **핵심 문제**: 사용자가 겪는 구체적 고통 (Pain point)
- **해결책**: 이 제품이 문제를 어떻게 해결하는지
- **차별화 포인트**: 경쟁 제품 대비 핵심 차이 (1-3가지)
- **성공 비전**: 6개월 후 이 제품이 달성할 모습

### 2. 시장 분석 (리서치 기반)
- **경쟁 제품 비교표** (표 형식)
  | 제품명 | 핵심 기능 | 강점 | 약점 | 가격 |
- **사용자 불만 패턴**: 경쟁 제품 리뷰에서 추출한 공통 불만 3가지
- **기회 영역**: 우리가 파고들 수 있는 틈새
- **기술 트렌드**: 활용할 수 있는 최신 기술/API

### 3. 타겟 사용자
각 페르소나(2-3개):
- **이름/역할**: 가상 인물 설정
- **나이/직업/기술 수준**
- **핵심 니즈**: 이 사람이 이 앱을 찾는 이유
- **현재 해결 방법**: 지금 어떻게 문제를 해결하는지
- **전환 동기**: 왜 우리 제품으로 전환하는지
- **사용 시나리오**: 일상에서 앱을 쓰는 구체적 상황

### 4. 핵심 기능 (우선순위 순)
각 기능마다:
- **기능명**
- **상세 설명**: 기능이 정확히 무엇을 하는지 (2-4줄)
- **사용자 스토리**: "~으로서, ~을 할 수 있다. ~을 위해"
- **우선순위**: P0(MVP 필수) / P1(중요) / P2(보완)
- **성공 지표**: 측정 가능한 수치 목표 (예: DAU 500+, 전환율 15%+)
- **인수 조건 (Acceptance Criteria)**: 이 기능이 "완료"인 조건 목록
  - [ ] 조건 1
  - [ ] 조건 2
  - [ ] 조건 3
- **에러 시나리오**: 실패할 수 있는 상황과 대응

### 5. 사용자 플로우 (핵심 5가지)
각 플로우:
- **플로우 이름**: (예: "신규 가입 플로우")
- **트리거**: 사용자가 이 플로우에 진입하는 계기
- **정상 경로 (Happy Path)**: 단계별 상세 (화면명 포함)
  1. [화면: 랜딩] 사용자가 CTA 버튼 클릭
  2. [화면: 가입폼] 이메일/비밀번호 입력
  3. ...
- **대안 경로 (Alternative Path)**: 정상이 아닌 다른 성공 경로
- **에러 경로**: 실패 시나리오와 에러 메시지
  - 네트워크 오류 → "연결을 확인하세요" 토스트
  - 중복 이메일 → 인라인 에러 "이미 등록된 이메일"
- **빈 상태 (Empty State)**: 데이터 없을 때 화면 구성
- **로딩 상태**: 로딩 중 표시 방법 (스켈레톤/스피너)

### 6. 데이터 모델 (초안)
핵심 엔티티와 관계 정의:
```
User
├── id: UUID (PK)
├── email: string (unique)
├── name: string
├── created_at: timestamp
└── relationships: has_many Posts

Post
├── id: UUID (PK)
├── user_id: UUID (FK → User)
├── title: string
├── content: text
├── status: enum(draft, published)
├── created_at: timestamp
└── relationships: belongs_to User, has_many Comments
```
- 각 엔티티의 필드, 타입, 제약조건 명시
- 엔티티 간 관계 (1:N, N:M) 명시

### 7. 화면 목록 및 구성
주요 화면 목록 (10-15개):
| # | 화면명 | 경로 | 핵심 요소 | 연결 화면 |
|---|--------|------|-----------|-----------|
| 1 | 랜딩 | / | 히어로, CTA, 기능 소개 | 가입, 로그인 |
| 2 | 로그인 | /login | 이메일/PW 폼, 소셜 로그인 | 대시보드, 가입 |
| ... | ... | ... | ... | ... |

각 화면의 핵심 컴포넌트와 인터랙션 설명 포함.

### 8. API 엔드포인트 (초안)
핵심 API 목록:
| Method | Path | 설명 | 요청 Body | 응답 |
|--------|------|------|-----------|------|
| POST | /api/auth/signup | 회원가입 | {email, password, name} | {user, token} |
| POST | /api/auth/login | 로그인 | {email, password} | {user, token} |
| GET | /api/posts | 게시물 목록 | - | {posts: [...]} |
| ... | ... | ... | ... | ... |

### 9. MVP 범위
- **기간**: 3주 (15영업일)
- **주차별 마일스톤**:
  - Week 1: [구체적 완료 항목]
  - Week 2: [구체적 완료 항목]
  - Week 3: [구체적 완료 항목]
- **포함 (In Scope)**: 항목별 나열
- **제외 (Out of Scope)**: 항목별 나열 + 제외 이유
- **V2 예정**: 향후 추가할 기능 목록

### 10. 비기능 요구사항
- **성능**: 초기 로드 < 2초, API 응답 < 200ms, LCP < 2.5초
- **보안**: 인증 방식, 데이터 암호화, OWASP Top 10 대응
- **접근성**: WCAG 2.1 AA 수준
- **국제화**: 지원 언어, RTL 고려
- **SEO**: 메타태그, OG 태그, 구조화 데이터
- **모니터링**: 에러 추적 (Sentry), 분석 (Analytics)
- **플랫폼**: Web (반응형) / iOS / Android / 데스크톱

### 11. PRD 자체 검증
작성 완료 후 아래 항목을 점검하고 결과를 PRD 하단에 포함:
- [ ] 모든 P0 기능에 인수 조건과 성공 지표가 있는가
- [ ] 사용자 플로우에 에러/빈/로딩 상태가 모두 정의되어 있는가
- [ ] 데이터 모델이 모든 핵심 기능을 커버하는가
- [ ] API 엔드포인트가 모든 화면 동작을 지원하는가
- [ ] MVP 범위가 3주 내 실현 가능한 수준인가
- [ ] 비기능 요구사항에 구체적 수치가 있는가
- [ ] 경쟁 분석 결과가 차별화 포인트에 반영되었는가
